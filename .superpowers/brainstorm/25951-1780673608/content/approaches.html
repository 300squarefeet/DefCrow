3 Pendekatan Implementasi

Bagaimana Rust loader di-generate dari pilihan user di web UI?

A

Tera Template Engine

Source code Rust loader disimpan sebagai file .rs.tera template. Backend Axum mengisi variabel (fitur yang dipilih, key enkripsi, config) lalu invoke cargo build. Output binary langsung distreaming ke user.

Pro

  • Template mudah dibaca dan dikembangkan
  • Fleksibel — tiap teknik OPSEC adalah blok template terpisah
  • Cocok untuk AppDomain .config generation

Kontra

  • Butuh Rust toolchain di server
  • Build ~30-60 detik per request
B

Cargo Feature Flags (Rekomendasi)

Setiap teknik OPSEC adalah Cargo feature (--features amsi,etw,sleep-encrypt,direct-syscall). Web app memanggil cargo build dengan kombinasi feature yang dipilih. Kode loader bersih dan idiomatic Rust.

Pro

  • Kode loader paling bersih — conditional compilation dengan #[cfg(feature=...)]
  • Mudah test tiap fitur secara independen
  • Build cache Cargo menghemat waktu untuk kombinasi yang sama

Kontra

  • Butuh Rust toolchain di server
  • Kombinasi fitur baru selalu rebuild dari awal
C

Pre-compiled Shellcode Stub + Runtime Assembly

Tiap teknik OPSEC dikompilasi menjadi shellcode stub terpisah. Web app menggabungkan stub yang dipilih + shellcode user pada runtime — tanpa memanggil compiler. Output langsung instan.

Pro

  • Generate instan — tidak butuh toolchain di server
  • Bisa deploy ke cloud tanpa Rust

Kontra

  • Kompleksitas tinggi — tiap stub harus position-independent
  • Debugging sulit
  • AppDomain + .config tidak cocok dengan model ini

Rekomendasi saya: B (Cargo Feature Flags) — kode loader paling bersih, Cargo build cache membuat request kedua jauh lebih cepat, dan mendukung semua fitur termasuk AppDomain + .config secara natural.
