Strategi Kecepatan Generate — Pilih Pendekatan

Bagaimana frontend berkomunikasi dengan Rust build server secara efisien?

A

Pre-compiled Scaffold + Thin Config (Rekomendasi)

Mirip cara ScareCrow kerja: semua modul OPSEC di-compile sekali jadi libscaffold.rlib. Tiap request hanya generate file konfigurasi tipis (~50 baris), lalu rustc config.rs --extern scaffold=libscaffold.rlib — link saja, tidak recompile semua modul.

Timeline
Server startup:  cargo build semua modul → libscaffold.rlib  (~90s, sekali)
Per request:     rustc config.rs --extern scaffold              (~5-12s)
Warm cache:      sama → instan dari sccache                    (~1-2s)

Pro

  • 5-12s per generate setelah warmup
  • Mirip pola ScareCrow (zip scaffold)
  • sccache cache tiap kombinasi

Kontra

  • Server startup ~90s (sekali)
B

Full Cargo Build + Incremental Cache (sccache)

cargo build tiap request, tapi sccache men-cache hasil compile tiap file. Kombinasi feature yang sama = instant dari cache. Kombinasi baru = rebuild hanya file yang berubah.

Timeline
Kombinasi baru:  cargo build --features X,Y,Z    (~45-90s)
Kombinasi sama:  dari sccache                    (~3-8s)
Pre-warm:        build semua kombinasi umum saat startup

Pro

  • Paling fleksibel
  • sccache bisa di-share antar instance

Kontra

  • Kombinasi baru tetap lambat ~90s
C

Pre-compiled Object Files + LLD Linker

Tiap modul OPSEC di-compile menjadi .o object file terpisah. Generate = pilih object files yang diminta + link dengan LLD. Paling cepat tapi paling kompleks.

Timeline
Server startup:  compile tiap modul → .o files  (~90s, sekali)
Per request:     lld link pilihan module          (~1-3s)
Semua request:   selalu cepat, tidak ada cache

Pro

  • 1-3s selalu, tidak perlu cache

Kontra

  • Rust object files tidak dirancang untuk linking manual
  • Feature flags tidak bisa dikontrol per-object

Komunikasi Frontend ↔ Backend — Semua Opsi Pakai Pola Ini

WebSocket + REST Hybrid
1. POST /api/generate (REST)
   → Validasi config, buat job_id
   → Return: { job_id: "abc123" }   ← INSTAN

2. WebSocket: ws://host/ws/jobs/abc123
   → Server stream progress real-time:
     {"status":"queued",    "progress":0,  "msg":"Waiting..."}
     {"status":"building",  "progress":15, "msg":"Generating source..."}
     {"status":"building",  "progress":40, "msg":"Compiling scaffold..."}
     {"status":"building",  "progress":85, "msg":"Linking & signing PE..."}
     {"status":"done",      "progress":100,"msg":"Ready","download_id":"xyz"}

3. GET /api/download/xyz (REST)
   → Stream binary file

Frontend UX:
   → Progress bar real-time dari WebSocket
   → Tidak ada polling — server push
   → Koneksi WS otomatis tutup saat done/error
